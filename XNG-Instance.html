Purpose

This document has a suggestion for an alternate representation of the XBRL syntax. The goal is to provide a method of communicating Burinss Reporting generally for financial regulations. The goal is sound but the historical implementation could be refreshed. I believe that adoption of XBRL is limited because of the complexity of the syntax more than the sophistication of the business user. By looking at a syntax refresh could we reach out more broadly then we are today and have XBRL become a defacto standard for a broader usage.

This document is specific to the XBRL instance syntax.

Introduction

The purpose of this document is to explain an alternate format to represent an instance document which parallels the current functionality.

The reboot is named XBRL-Next Generation (aka "xng"). The goal is to three fold:

  • to maintain the business functionality of the current XBRL 2.1 or the XBRL Dimensions 1.0 specifications
  • to revisit the syntactic implementation to decouple unnecessarily linked aspects of the technology.
  • to make XBRL less complex in the technical marketplace without reducing the business functionality which it describes well.

    • A part of this work has follow on from my previous efforts as a member of the XBRL Abstract model task force. The logical model we developed was much simpler than the XBRL file format. As an IT professional looking at the complexity of the XBRL Syntax as compared to the Abstract model there is a logical step to ask the question "Can we improve on where we are?"

    What does XBRL do?

    XBRL is a platform for articulating semantic information specific to financial reporting. There are mandatory modeling requirements to ensure that the financial information is communicated correctly.

    The costs of using XBRL at a technical level are very high. Both in tool investment and in the amount of knowledge require to fully appreciate the level of technical detail required to design a taxonomy or to articulate a filing. I believe that the implementation choices such as the way in which XML schema is used and Xlink is used have resulted in a technically complex taxonomy and instance design.

    I have replaced syntax such as contexts, segment, and scenario and have tried to relate the network of information more closely which has resulted in a smaller core schema structure. From which the network of Facts can be inferred.

    Instance Meta Model

    A Fact will be related to other Facts as Facets. A Fact will have a second relationship to other Facts by Groups.

  • A Document will contain a number Facts.
  • Identity fields, Date fields and dimensions fields are all unique Facts.
  • Facts will be related directly to Facts by Facet Relationships.
  • Identity fields, Date fields and dimensions fields are not mandatory by Scheme design. They can be mandatory based on a secondary parser in the same was could be described by XBRL Formula or XML Schematron XSD verifier.
  • A Fact can be related to another Fact by a Grouping relationship. This is a secondary part of a Facet. This can be used to Group sets of repeating fields.

    What could it look like?

    Two large sets that are applied to the two Facts

    This example is to show a sample instance that is similar in structure to the current XBRL instance document structure. An example of the traditional structure (at traditional00) is provided afterwards to parallel. This will show the idea of a Facet and Facet reference that will be further explained as the document progresses.

    In this first example you can easily see that EntityType, EntityValue, StartDate, EndDate and Unit plus the two dimension references Region and EmployerType are set up in a Facet object. This Facet object, to a degree, aligns with the context and context ref of the XBRL instance schema. Segment and Scenario have been removed and the Unit is also inside the Facet.

    The xng namespace applied to EntityType, EntityValue, StartDate, EndDate and Unit references these as predefined resources in the new XNG schema.

    All elements (EntityType, EntityValue, StartDate, EndDate, Unit, Region, EmployerType, Taxable.Amount and Taxable.Code) derive from the same schema base type called "xng:baseObject".

    The xng:baseObject has a value attribute which is always string and also has the ability to have a child which is also of type xng:baseObject.

    The model of the basetype element relating to another baseObject element results in a cyclic model where the data can be turned around a number of ways to produce the same result.

    The presentation of example00 as the first example was to show a similar to the current XBRL instance syntax. This model though can turn the information around in a number of ways that all result in the same set of relationships.

    Each of the Context parts EntityType, EntityValue, InstantDate, StartDate and EndDate and separately Unit will be referred to as Facets of the Fact. As long as the Fact can discover the Facet in its hierarchy (either above or below or alongside) then that Facet will be linked the Fact.

    The purpose of the Facet and Facet ref element is not to concretely abstract the Facet and dimensional values from the Fact but to provide a syntax within the instance document to allow for sets of value can be set and reused. These Sets are linked directly to the Fact and when parsed the sets are removed resulting in a direct relationship between the each base object with each related base object.

    The articulation of the Fact is made up of any hierarchically traversable parent of any traversable child.

    Facts with Facets as children

    These two Facts differ from example00 by demoralizing the Facet value into direct relationships of the Fact.

    Following the cyclic model the base object can refer to any other base object. Just as long as all of the conceptualizing Facets and dimension elements are discoverable either as an ancestor or as a child the conceptualization of the Fact is complete.

    Articulating each Facet as a child aspect of the Fact allows for the full details of the Fact to be expressed in a simple form.

    When Streaming large XML documents there is no longer a document ordering issue as the demoralized Fact is now complete when it occurs.

    Fact Groups (Siblings)

    The Facet model above is focused on connecting facts to facts via a facet relationship. Facts though can have a second relationship to other Facts through sibling relationships. This sibling relationship is saying that the fact should be understood as a part of the group it is within.

    An easy example of a fact to fact relationship is in an group of addresses. When a Residential and Postal address is to be reported against a participant there needs to be a way to explicitly link the Building number, Street Name, Suburb, State and Country to each other and to the Postal or Residential address concept.

    When both Postal or Residential address are in the same payload the grouping of the objects needs to be distinguished from the fields in both groups.

    When there are multiple instances of a set the relationship between those groups should also be clear.

    Instance Schema

    Here is the first look at the Schema which underpins the document structure. Basic by design. Easily parsable. No specific knowledge of the final document structure.

    The specific datatype on the value attribute can be parsed by inference. Following the desire to decouple document structure from business rules the base schema will not have a specific data type for an element. This will be able to be set externally in a different XSD and linked into a parser via a resource file.

    The result of this decoupling is that the change to the valid enumerations of a Fact does not need to change the underlying document structure. Reducing the interconnectedness resulting in faster and more isolate changes. For the implementing software developers both receiving and authoring this can reduce the costs to implement.
